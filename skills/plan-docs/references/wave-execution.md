# 波次执行规则

## 1. 从依赖图生成 Wave

1. 为每个任务分配稳定 ID。
2. 用 `depends_on` 建立有向无环图。
3. 无未完成前置依赖的任务进入最早 Wave。
4. 同一 Wave 中只有彼此无依赖的任务才可能并行。
5. 有循环依赖时先修改方案，不得生成伪顺序。

## 2. 并行安全

同时满足以下条件才能并行实现：

- `owned_paths` 不重叠。
- 不修改同一接口、schema、配置键或共享状态。
- 不依赖另一任务尚未产生的输出。
- 每项都有独立验收方法。
- 中间状态不会破坏运行或部署。

文件不重叠不代表运行时安全。数据库、环境变量、权限、队列、API 契约和发布顺序也要纳入依赖分析。

不能证明安全时，降级为 `main` 或 sequential。

## 3. 执行者选择

| 执行者 | 适用范围 | 约束 |
|---|---|---|
| `main` | 单一短链路、共享文件、集成与收口 | 父代理直接完成 |
| `subagent:explore` | 跨模块阅读、未知调用链、兼容性和方案调查 | 只读；调用 `Task` 时使用 `subagent_type="explore"`、`readonly=true`，不修改文件 |
| `subagent:generalPurpose` | 独立模块实现或独立验证 | 调用 `Task` 时使用 `subagent_type="generalPurpose"`、`readonly=false`；普通实现选 Terra，高难度任务可选 Sol，并按需要设置 `thinking_effort`；必须声明 `owned_paths`、实际落盘并完成验收 |

## 4. Wave 生命周期

- `pending`：准入条件尚未满足。
- `ready`：所有依赖已完成。
- `in_progress`：当前正在执行。
- `blocked`：必需任务失败、超时、冲突或结果不明。
- `completed`：所有任务验收通过且完成门通过。

只允许最早未完成 Wave 进入 `in_progress`。

## 5. 完成门

完成门必须是可观察条件，例如：

- 指定构建或现有检查通过。
- 关键接口、状态或 UI 行为符合预期。
- 所有 Task 结果已返回并被父代理综合。
- 没有未说明的文件冲突或兼容风险。

不得使用“代码已写完”“Task 已启动”作为完成门。

## 6. 失败与恢复

- 单个任务失败时保留已完成任务，不重复派遣。
- 失败任务修复或 resume 后重新验收。
- 运行中状态在重启后无法证明完成时标记 blocked，不静默重派。
- 恢复时按任务 ID、Todo 状态和计划文本确定当前 Wave。

## 7. 原子性交付

只有以下情况使用 `must land together`：

- schema 与读取代码必须同时上线。
- 移除旧配置与切换新配置不可分割。
- 中间版本会导致数据损坏、鉴权失效或服务不可用。

其他情况优先 expand-contract 或普通顺序 Wave。
